Documento de Arquitectura de Solución

Sistema: Sentimientos_X · Versión 2.0 · Pipeline Inteligente

Arquitectura desacoplada, escalable y preparada para análisis de sentimiento en tiempo real sobre datos de X (Twitter).

1. Definición de la Arquitectura

La solución implementa una arquitectura basada en eventos y microservicios ligeros con un pipeline de datos asíncrono. La extracción, procesamiento NLP, almacenamiento y visualización están desacoplados, permitiendo cambiar el foco de análisis (temas, cuentas o hashtags) modificando únicamente el archivo config.yaml. Este enfoque garantiza adaptabilidad, resiliencia y facilidad de mantenimiento, con un núcleo central en SQLite optimizado para retención de 30 días.

2. Recursos Disponibles

  • Hardware: Servidor Cloud con 2 vCPU, 4GB RAM (escalable) o estación de trabajo local (Python 3.10+).
  • Software Base: Ubuntu 22.04 LTS / Windows 11 + WSL2, entornos virtuales Python.
  • Conectividad: API v2 de X (Bearer Token, nivel Essential o Basic).
  • Almacenamiento: SQLite con cifrado opcional + backups automáticos.
  • Orquestación: Cron jobs o GitHub Actions para ejecución programada.

3. Seguridad y Privacidad

Estrategia de cumplimiento y protección de datos alineada con estándares internacionales y Ley 25.326 (Argentina).

  • Credenciales seguras: Gestión mediante variables de entorno + Vault local (Hashicorp Vault simplificado).
  • Anonimización: Eliminación de metadatos personales antes del análisis, almacenando solo ID del post y sentimiento agregado.
  • Auditoría: Logs cifrados de acceso a datos y registros de operaciones críticas.
  • Retención limitada: Política de purga automática para datos > 30 días.

6. Patrones de Diseño Aplicados

  • Singleton (Database Pool): Control de conexión única a sentimientos.db para evitar race conditions.
  • Adapter (API Wrapper): Adaptador entre respuestas crudas de X y objetos de dominio homogéneos.
  • Observer (Pipeline Notifications): Disparadores automáticos para actualizar dashboards cuando se ingieren nuevos datos.
  • Strategy (Visualización): Permite cambiar entre gráficos de barras, líneas temporales o nubes de palabras sin modificar lógica central.

8. Plan de Pruebas (QA Estratégico)

  • Pruebas Unitarias (pytest): Validación de función de polaridad, conexión mockeada a API y lógica de deduplicación.
  • Pruebas de Integración: End-to-end desde extracción hasta escritura en SQLite y generación del dashboard HTML.
  • Pruebas de Rendimiento: Simulación de rate limits (450 requests cada 15 min) y volumen de 10k tweets.
  • Pruebas de Regresión: Ejecución en entorno CI/CD cada vez que se modifica el pipeline.
